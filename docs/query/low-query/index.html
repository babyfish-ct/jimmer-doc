Low Query

Low Query is a KSP-based helper for Kotlin projects. It reads query annotations on Jimmer entities and generates typed query functions, so repetitive back-office filter pages can share the same dynamic where and orderBy rules without writing boilerplate for every controller.

It is not a replacement for the normal SQL DSL. Use the normal DSL when each endpoint has its own query shape. Use Low Query when many list pages follow the same pattern: optional equality filters, keyword search, ranges, in filters, and stable default ordering.

KSP configuration​

Low Query generates Kotlin extension functions, so it is a KSP-only processor. It does not provide a Java APT processor.

Use a KSP plugin version compatible with your Kotlin version:

plugins {
    kotlin("jvm")
    id("com.google.devtools.ksp")
}

dependencies {
    implementation("org.babyfish.jimmer:jimmer-low-query-annotations:<jimmer-version>")
    ksp("org.babyfish.jimmer:jimmer-low-query-processor:<jimmer-version>")
}

The KSP option is optional. The generated package defaults to <entity-package>.generated.lowquery; override it only when you want all generated query helpers in one package:

ksp {
    arg("jimmerLowQuery.generatedPackage", "com.example.generated.lowquery")
}

For a complete Gradle Kotlin DSL setup:

plugins {
    kotlin("jvm")
    id("com.google.devtools.ksp") version "<ksp-version>"
}

dependencies {
    implementation("org.babyfish.jimmer:jimmer-sql-kotlin:<jimmer-version>")
    implementation("org.babyfish.jimmer:jimmer-low-query-annotations:<jimmer-version>")
    ksp("org.babyfish.jimmer:jimmer-ksp:<jimmer-version>")
    ksp("org.babyfish.jimmer:jimmer-low-query-processor:<jimmer-version>")
}

ksp {
    arg("jimmerLowQuery.generatedPackage", "com.example.generated.lowquery")
}
note

Keep using the normal Jimmer KSP processor (jimmer-ksp) for entity code generation. Low Query is an additional KSP processor that generates query helpers from the entity model.

Add annotations to entities​

import org.babyfish.jimmer.lowquery.annotation.Eq
import org.babyfish.jimmer.lowquery.annotation.Keyword
import org.babyfish.jimmer.lowquery.annotation.Like
import org.babyfish.jimmer.lowquery.annotation.OrderByDesc
import org.babyfish.jimmer.sql.Entity
import org.babyfish.jimmer.sql.Id

@Entity
interface Book {
    @Id
    @OrderByDesc
    val id: Long

    @Like
    @Keyword
    val name: String

    @Eq
    val storeId: Long?
}

For the entity above, the processor generates a KMutableRootQuery.ForEntity<Book> extension. A list endpoint can call it inside a regular Jimmer query:

import demo.generated.lowquery.query

fun findBooks(
    name: String?,
    storeId: Long?
): List<Book> =
    sqlClient
        .createQuery(Book::class) {
            query(name = name, storeId = storeId)
        }
        .execute()

Nullable parameters are skipped by dynamic predicates. In the example above, name = null does not add the name condition.

Query operators​

Low Query supports the following annotations:

AnnotationGenerated condition
@Eq / @Neequality / inequality
@Likeilike(..., LikeMode.ANYWHERE)
@StartsWith / @EndsWithprefix / suffix like
@Gt / @Ge / @Lt / @Lecomparison
@In / @NotIncollection membership
@Betweenlower and upper bound parameters
@TimeRangestart and end time parameters
@Keywordcontributes a field to the shared keyword OR predicate
@OrderByAsc / @OrderByDescgenerated orderBy expression
@JimmerFindByextra findBy... helper functions

@JimmerLowQuery can be placed on the entity to customize generated function names, visibility, and fetcher strategy.

Implicit equality fields​

If an entity has at least one Low Query annotation, scalar business properties without explicit query annotations are generated as nullable equality parameters automatically.

Infrastructure fields are excluded from this implicit rule unless they are explicitly annotated. Typical excluded fields include id, createTime, updateTime, creator, creatorId, updater, updaterId, deleted, deletedTime, and tenantId.

This lets entities mark only the fields that need special behavior:

@Entity
interface Device {
    @Id
    val id: Long

    @Like
    @Keyword
    val name: String

    @TimeRange(name = "createTime")
    val createTime: LocalDateTime

    val status: Int?
}

In this example, name uses fuzzy search, createTime uses a time range, and status is still generated as a nullable equality parameter.

Query by loaded entity properties​

The processor also generates a KSqlClient.createLowQuery(entity) extension. It checks loaded properties through ImmutableObjects.isLoaded(entity, "property"), so a caller can pass a Jimmer draft containing only fields that should participate in filtering:

val probe = Book {
    name = "GraphQL"
    storeId = 1L
}

val books = sqlClient.createLowQuery(probe).execute()

Only loaded properties are translated into query conditions.

Runtime provider for generic controllers​

When Spring @Component is available on the compilation classpath, the processor also emits one JimmerLowQueryProvider bean for each entity. Generic controller code can inject providers and apply low-query rules without importing generated extension functions from each entity package.

The provider exposes:

  • the entity type;
  • the entity-property to request-parameter name mapping;
  • whether the generated query writes ordering;
  • apply(query, entity) for entity-probe filtering;
  • apply(query, source) for parameters that cannot be represented by a draft entity, such as range and collection parameters.
